# Meridian — Architecture

> Scope note: this architecture describes a local/demo portfolio system using synthetic sample data only.

## 1) System topology

```text
Frontend (React/TypeScript)
        |
        | REST + JWT
        v
Backend API (FastAPI routers)
        |
        +--> Scoring engine (rules + logistic regression baseline)
        +--> Explainability service (SHAP contributions)
        +--> Review workflow (queue/assignment/decisions/history)
        +--> Metrics + model evaluation endpoints
        |
        v
Persistence (PostgreSQL via SQLAlchemy)
```

## 2) Core architecture capabilities

- **Transaction intake and scoring** for near-real-time risk assessment.
- **Review queue operations** for analyst triage and override decisions.
- **Explainability retrieval** for transaction-level factor interpretation.
- **Model evaluation services** for synthetic benchmark reporting.
- **Simulation-driven demo mode** for reproducible interview walkthroughs.

## 3) Fraud scoring workflow (backend perspective)

1. Transaction payload validated by API schema.
2. Features generated by shared extraction logic.
3. Rule layer computes deterministic risk indicators.
4. Baseline classifier computes fraud probability.
5. Decision band assigned (`approve/review/decline`).
6. Decision trace and reason codes persisted.
7. Explainability data served on demand.

This pipeline supports interview conversations about training/serving alignment and auditable decision paths.

## 4) Review queue architecture

Review lifecycle components:

- **Queue listing**: status-filtered pending/resolved workloads.
- **Assignment**: analyst/reviewer ownership transitions.
- **Decisioning**: approve/decline/escalate outcomes with notes.
- **History**: chronological state-change timeline.
- **Auditability**: actions persisted for governance demos.

## 5) Explainability architecture

Explainability claims are tied to implemented behavior:

- SHAP contribution values for per-transaction feature impact.
- Top-factor extraction for analyst UX.
- Narrative summaries for faster interpretation.

No claims of causal inference or regulatory sufficiency.

## 6) Model evaluation architecture

Evaluation can be triggered through offline scripts and API endpoints:

- Classification metrics (precision, recall, F1, ROC-AUC)
- Calibration indicator (Brier score)
- Confusion matrix and threshold-sensitive tradeoffs
- Cost-oriented framing for fraud operations discussion

All reported results are synthetic-data demonstrations.

## 7) Security and role design

- JWT-based authentication
- RBAC roles (Admin/Analyst/Reviewer/Viewer)
- Route-level role checks for sensitive operations

## 8) Responsible use and non-goals

- No real banking data pipelines.
- No production incident response guarantees.
- No claims of real-world fraud-loss prevention efficacy.
- Intended for portfolio demonstration, learning, and technical interviews.
